### Issue #1 — TenantMiddleware não suporta autenticação directa por API Key

**Área:** Backend — `apps/api/src/common/tenant-context/tenant.middleware.ts`
**Severidade:** Alta
**Ambiente:** Local + Produção

**Descrição:**
O `TenantMiddleware` extrai o `tenantId` apenas de dois lugares:
- Header `Authorization: Bearer <jwt>` — decode do JWT
- Header `x-tenant-id` — valor directo

Quando um sistema externo autentica via `x-api-key` sem JWT, o middleware não consegue identificar o tenant e retorna 401 antes de chegar ao `ApiKeyStrategy`.

**Comportamento actual:**
```bash
curl -H "x-api-key: <valid-key>" /api/events
# → 401 Tenant not identified
```

**Comportamento esperado:**
```bash
curl -H "x-api-key: <valid-key>" /api/events
# → 201 Created
```

**Causa raiz:**
O middleware não faz lookup na BD pelo hash da API Key para extrair o `tenantId`.

**Fix sugerido:**
Injectar `ApiKeyService` no `TenantMiddleware` e, quando não houver JWT, fazer `apiKeyService.validate(rawKey)` para obter o `tenantId` da key.

### Issue #2 — Rate limiting por entidade não funciona em produção

**Área:** Backend — `apps/api/src/modules/event/event.service.ts`
**Severidade:** Média
**Ambiente:** Produção (Railway)

**Descrição:**
O rate limiting por entidade usa Redis com chave `rate:{entityId}` e limite `MAX_EVENTS_PER_MINUTE=30`. Em produção, 31+ eventos são criados sem retornar 429.

**Comportamento actual:**
```bash
# 31 requests consecutivos — todos retornam 201
```

**Comportamento esperado:**
```bash
# Request 31 → 429 Too Many Requests
```

**Variável confirmada no Railway:** `MAX_EVENTS_PER_MINUTE=30` ✅

**Causa raiz provável:**
O `CacheManager` com `@keyv/redis` pode não estar a persistir correctamente os contadores TTL em produção, ou há incompatibilidade na forma como o TTL é definido com o Keyv adapter.

**Fix sugerido:**
Adicionar logging do contador Redis no `EventService` para diagnóstico, e verificar se as keys `rate:*` estão a ser criadas no Redis de produção via Railway Redis CLI.

### Issue #3 — ThrottlerModule global não funciona em produção

**Área:** Backend — `apps/api/src/app.module.ts`
**Severidade:** Média
**Ambiente:** Produção (Railway)

**Descrição:**
O `ThrottlerModule` está configurado com limite de 30 requests por minuto por IP. Em produção, 35+ requests consecutivos passam todos com 201 sem 429.

**Comportamento actual:**
```bash
# 35 requests consecutivos — todos retornam 201
```

**Comportamento esperado:**
```bash
# Request 31+ → 429 Too Many Requests
```

**Causa raiz provável:**
O `ThrottlerModule` usa armazenamento em memória por defeito. Em produção com Railway Edge Proxy, o IP real do cliente não chega ao NestJS (chega o IP do proxy), fazendo com que todos os requests pareçam vir de IPs diferentes. Alternativa: o storage em memória é resetado entre deploys/restarts.

**Fix sugerido:**
Configurar `ThrottlerModule` com storage Redis em vez de memória, e configurar `trust proxy` no NestJS para extrair o IP real via header `X-Forwarded-For`.

### Issue #4 — Frontend não envia apiKey no pedido de autenticação

**Área:** Frontend — `apps/web/src/services/api.ts`
**Severidade:** Alta
**Ambiente:** Local

**Descrição:**
O `authService.getToken` envia apenas `{ clientId }` para `POST /api/auth/token`. Após o Sprint 16, o endpoint exige `{ clientId, apiKey }`. O dashboard retorna "ERRO: Falha na ligação ao servidor."

**Código actual:**
```typescript
getToken: async (clientId: string) => {
  const { data } = await api.post('/auth/token', { clientId });
  return data.accessToken as string;
},
```

**Código esperado:**
```typescript
getToken: async (clientId: string, apiKey: string) => {
  const { data } = await api.post('/auth/token', { clientId, apiKey });
  return data.accessToken as string;
},
```

**Causa raiz:**
O `api.ts` não foi actualizado quando o `AuthController` e `AuthService` foram refactorizados no Sprint 16.

**Fix sugerido:**
1. Actualizar `authService.getToken` para aceitar e enviar `apiKey`
2. Definir `NEXT_PUBLIC_API_CLIENT_ID` e `NEXT_PUBLIC_API_KEY` no `.env.local`
3. Actualizar todas as páginas que chamam `authService.getToken`
4. Avaliar se as credenciais devem vir de variáveis de ambiente ou de um fluxo de login real
